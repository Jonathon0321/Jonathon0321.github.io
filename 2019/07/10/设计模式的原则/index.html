设计模式的原则

Jul 10, 2019

设计模式有23种，这23种模式都遵守六个原则。六个原则分别是单一职责原则、里氏替换原则、依赖倒置原则、接口隔离原则、迪米特法则和开闭原则。

一、单一职责原则

​ 该原则的原文是：”There should never be more than one reason for a class to change“，由罗伯特·C.马丁（Robert C. Martin）提出。对于一个类，使其发生改变的原因不应该超过一个。这里的原因即是职责的意思，即每个类应只有一种职责。
​ 一种职责指的是一个或多个紧密相关的功能。例如，作为播放器，播放、暂停等功能是不可分割的，而上传、下载等功能就应该放到一个新的类中，而不是草率的加入到播放器中。
​ 该原则的判断基准是职责，而职责却是难以量化的，因此一个设计是否严格遵守了单一职责原则，是难以下定论的。在实际设计中，如何细分职责，细分之后是否每一个都单独成一个接口，都需要和实际结合，如果一味的遵守原则，设计出来的接口反而可能过于繁多，难以使用。

二、里氏替换原则

​ Liskov在1987年提出了一个关于继承的原则：

​ “Inheritance should ensure that any property proved about supertype objects also holds for subtype objects.”——“继承必须确保超类所拥有的性质在子类中仍然成立。”

​ 在该原则中，对该原则的解释共有四项：

  1. 子类必须完全实现父类的方法
  2. 子类可以有自己的个性
    1. 覆盖或实现父类方法时输入参数可以被放大
    2. 覆写或实现父类的方法时输出结果可以被缩小

三、依赖倒置原则

​ 依赖倒置原则的定义如下：

High level modules should not depend upon low level modules,both should depend upon abstractions.Abstractions should not depend upon details.Details should depend upon abstracts.

​ 该原则翻译后如下：

  • 高层模块不应依赖于低层模块，两者都应依赖于抽象
  • 抽象不应依赖于细节
  • 细节应该依赖于抽象

​ 高层模块和底层模块比较好区分，具有原子性、不可再分的模块即为最低层的模块，由底层模块进行组装构成的即为高层模块。抽象在JAVA中指的是接口或者抽象类，这二者都不可被实例化；而细节指的就是接口的实现和继承了抽象类的那些类 ，这二者都可以被实例化。

​ 何为倒置？正常的依赖应该是这样的：我要去开奔驰车，我就依赖于奔驰车；我需要去开宝马车，那我就去依赖宝马车。可是这些一旦放到程序中则行不通了，如果真的按照这种依赖，拿将需要些大量重复代码。在软件工程中，往往是对现实进行抽象，然后再进行设计。将宝马车和奔驰车都进行抽象可以得到一个汽车的抽象类，这个抽象类包含汽车的全部特性，这时司机就可以依赖于汽车类，实现时根据情况去调用具体的汽车类（例如宝马、奔驰）。同时这样的设计还拥有很强的扩展性，如果将来还有其他的车需要开，则直接继承汽车类并实现其内部功能即可，并不需要去修改司机对应的类，大大减少了工作量。这就是依赖倒置。

​ 根据依赖倒置原则，我们不应依赖具体的类，而是所有都依赖于接口或抽象类，而在实际开发中，很难完全遵守这个原则，例如，对与Utils类，无需抽象类或接口，这时候就不必拘泥于依赖倒置原则。
